# xfstests baseline: generic/* tests that ACTUALLY PASS against the clio FUSE
# filesystem (TEST_DIR-only harness). CI runs ONLY these via
# scripts/xfstests/run_ci_xfstests.sh and every one must pass.
#
# 2026-07-05 INTEGRITY REBUILD. This list was rebuilt from 513 -> 81 entries
# after finding that scripts/xfstests/run_clio_xfstests.sh miscounted NOTRUN
# tests as PASS (it grepped ./check output for "^Passed all" before "^Not run:";
# a notrun test prints BOTH "Not run: <t>" and "Passed all 0 tests"). A prior
# bulk sweep had therefore admitted ~430 tests that never actually ran -- mostly
# _require_scratch (SCRATCH_DEV is intentionally unset on this TEST_DIR-only
# harness, so they notrun in EVERY environment; 130 of them) plus tests needing
# an xfs_io command / feature clio's fs does not provide (~300). run_ci treats
# any non-pass (incl. notrun) as failure, so with the harness bug fixed those
# entries would (correctly) fail CI. The harness fix is commit b0a616c3.
#
# The 81 tests below were verified passing across TWO independent runs with the
# corrected harness (2026-07-05). Scratch-requiring tests that pass under the
# two-runtime driver belong in ci_scratch_baseline_pass.txt, not here. Removed
# tests are recorded in the commit + scratchpad GATE_INTEGRITY_FINDING.md; any
# that genuinely pass (e.g. once a missing xfstests helper like src/dbtest is
# built) can be re-added after verification with the fixed harness.
#
# Slow-lane note: 208/323 (aio-dio) PASS but exceed the 90s per-test CI cap by
# design; keep them out of this gate (a longer-cap lane), or CI "hangs".
#
# 2026-07-05 CI-CPU FIX + quarantine. This gate went red in CI (run 28747864260):
# 12 tests (006/007/011/013/089/100/113/127/286/363/438/471) that pass on a
# 16-core dev box HUNG in the deps-cpu docker on the GitHub runner. Root cause:
# the embedded FUSE runtime's default 4 worker threads (5 busy-spinning workers)
# oversubscribe the runner's cores and livelock (reproduced by pinning to 2 CPUs
# locally). Capping the embedded runtime to num_threads=1 (in
# scripts/xfstests/clio_xfstests_config.yaml) RECOVERED HALF: verified in CI run
# 28751210007 the gate went 12 hangs -> 6 (006/007/100/113 pass again; 127 was
# already removed). The remaining SIX — 011/013/089/286/438/471 — still hang in
# CI's docker even at num_threads=1 and are QUARANTINED here.
#
# These six are genuine deadlocks, NOT slow-timeouts: CI timing shows they hit
# the 90s cap (~91s) while HEAVIER fsx tests pass fast in the same run (075=21s,
# 091=34s), so a bigger cap would not help. They pass on dev hardware; they
# deadlock only under CI's constrained docker scheduling — a self-send /
# lost-wakeup in the runtime's busy-spin worker model that a config knob cannot
# fix (num_threads=1 both removes oversubscription for some tests and, with a
# lone compute worker, can starve a task's own sub-task for others). The real
# fix is a runtime change (yield in ProcessNewTasks/BatchManager::FlushDue, or
# resolve the self-send/lost-wakeup) tracked in REMAINING_WORK.md. generic/127
# (fsx torture) is also out — it passes on >=4 cores but is borderline under a
# tighter cap. Gate: 84 -> 77 (127 + the six). Re-add each as the runtime fix
# lands and it passes two clean CI runs.
